# cc-suite：三个编程 Agent，能不能真正共用一套项目规则？

[主持人]
今天我们继续做“一份研究对应一期播客”。这期只讲一个仓库，李笑来的 `xiaolai/cc-suite`。它听起来像一个工具合集，但它真正想解决的，是很多人刚开始同时使用 Claude Code、Codex 和 Google 的 Antigravity 之后才会发现的问题：同一个项目，在三个 Agent 眼里可能根本不是同一个项目。

Claude Code 看 `CLAUDE.md`，Codex 看 `AGENTS.md`；Claude 的 Skills 可能在 `.claude/skills`，Codex 和 Antigravity 又去 `.agents/skills`；MCP 配置一边是 JSON，一边是 TOML。刚开始复制粘贴就能应付，过几周以后，规则改了一份忘了另一份，Agent 就开始各说各话。

所以我们今天不只问 cc-suite 有多少命令，而是问三个更实际的问题。第一，它到底怎样避免配置漂移？第二，它的双向委派是不是越多越好？第三，对我们现在的 GitHub 研究播客、视频生产和飞书自动化，最小应该借鉴哪一部分？

[分析员]
先给结论。cc-suite 可以理解成多编程 Agent 的翻译层和接线板。它不是一个新模型，也不是独立 IDE。它做的事情，是在项目文件层面建立共同来源，再把不同工具需要的格式转换出来；同时通过 MCP，让 Claude 能把任务交给 Codex，也让 Codex 能把任务交给 Claude。

本次研究固定在提交 `fbcf751`。仓库使用 ISC 许可证，插件清单版本是 `0.9.0`。它在 2026 年 5 月才创建，目前规模不大，也没有正式 GitHub Release，所以它更像一个值得小范围验证的快速演进项目，而不是可以无脑装进所有生产环境的基础设施。

它最核心的决定，是把 `AGENTS.md` 设为共享说明的源头。Claude Code 侧不再长期维护一份平行内容，而是让 `CLAUDE.md` 只保留一个 `@AGENTS.md` 导入。Codex 和 Antigravity 原生读取 `AGENTS.md`。这样，项目目标、完成标准、禁止事项这些共享意图，就不需要维护三份。

[主持人]
这里我有个疑问。既然要统一，为什么不干脆把 Claude Rules、Codex Rules、Subagents 和所有配置都统一？看上去不是更省事吗？

[分析员]
这恰恰是 cc-suite 做得比较克制的地方。它没有把“统一”理解成“全部复制”。README 明确列出不桥接的部分。Claude 的 `.claude/rules` 和 Codex 的 Rules 语义不同；Subagents 的 schema 和安全字段也不同。它选择把跨工具稳定的共享意图放进 `AGENTS.md`，而把平台专属能力留在各自边界。

这个设计比文件同步本身更值得借鉴。多平台治理应该是“共同规则加平台差异”，不是强行求同。否则为了让三边看起来一致，反而会丢掉每个平台真正重要的权限和安全语义。

Skills 的处理也类似。`scripts/bridge_skills.sh` 会先把 cc-suite 自己的 Skills 暴露到 Claude 的项目 Skills 目录，再建立 `.agents/skills` 指向 `.claude/skills` 的符号链接。结果是 Skill 内容只维护一份，Claude、Codex 和 Antigravity 能从各自认识的路径找到它。

但脚本不是见到同名路径就覆盖。真实目录已经存在时，它会停下来；只有路径安全时才创建链接。这种保守行为非常重要。自动化不是第一次运行成功就够了，还必须保证第二次、第五次运行不会把用户自己的目录改坏。

[主持人]
除了说明文件和 Skills，Hooks 与 MCP 是不是更麻烦？因为那不只是文本路径，里面还可能有命令、环境变量和凭据。

[分析员]
对。cc-suite 没有把两边的配置文件简单复制过去，而是做转换。`scripts/bridge_hooks.py` 只处理两边语义兼容的 Hook 事件。`scripts/bridge_mcp.sh` 会读取 Claude 项目的 `.mcp.json`，再生成 Codex 的 `.codex/config.toml` 区块，以及 Antigravity 的 `.agents/mcp_config.json`。

这里最值得注意的是“来源标记”。它写入自己管理的配置区块时会带 sentinel，也就是边界标记。以后更新只重写自己拥有的部分。遇到用户手动维护的同名 MCP 配置，它不会自作主张接管。这相当于告诉系统：机器生成内容必须标明所有权，否则未来没人知道哪几行能自动更新、哪几行属于用户。

Antigravity 的生成配置默认被忽略，因为里面可能包含凭据或机器专属路径。这也说明跨工具同步不是越彻底越好。能同步的结构和不该提交的秘密必须分开。

[主持人]
接下来谈最吸引人的部分：Claude 调 Codex，Codex 再调 Claude。这个听起来很像一个 Agent 团队。实际是怎么接起来的？

[分析员]
Claude 到 Codex 的方向，利用 Codex CLI 自带的 `codex mcp-server`。cc-suite 提供 `codex-runner.mjs`，负责启动进程、读取 Codex 的 JSONL 事件、记录任务状态、处理超时、后台运行和继续任务。于是 Claude 里可以使用审核、实现、故障分析、验证等命令，把边界明确的任务交给 Codex。

反方向则通过 `claude-octopus`。`scripts/mcp_claude.sh` 把它注册进 Codex 的 MCP 配置，Codex 可以调用 Claude 做 review、plan、implement 或 debug。项目固定了运行时依赖版本，避免每次 `npx` 拿到不同版本，但首次运行依然涉及网络下载，因此供应链和网络可用性不能忽略。

真正成熟的地方不是“能相互调用”，而是 Runner 还要回答谁在运行、任务 ID 是什么、日志在哪里、何时超时、结果有没有回来。多 Agent 最大的问题从来不是发出一句提示词，而是出了问题后还能不能追责和恢复。

[主持人]
那 Google 的 Antigravity 呢？它和 Codex 的接口能力似乎不完全一样。

[分析员]
差异很大。仓库说明里写得很清楚：`agy` 没有独立的 reasoning-effort 参数，推理强度被写在模型名称里；它也没有机器可读输出，命令主要打印普通文本；会话 ID 还要通过运行前后对本地会话目录做差异比较来恢复。

如果两个 `agy` 任务同时结束，新出现的会话可能不止一个。cc-suite 的策略不是猜，而是放弃记录会话 ID。这个细节非常值得我们学习。自动化遇到无法可靠关联的数据时，空缺比错误绑定更安全。空缺可以重试，错误绑定会让下游在错误上下文上继续做决定。

[主持人]
听到这里，感觉这个项目和我们现在的工作方式很接近。我们已经有 Codex Skills、长期自动化、GitHub 每日研究播客，还有飞书和 AI 视频生产。具体帮助在哪里？

[分析员]
第一个帮助，是给这些工作流建立共同的控制面。以现在的 GitHub 研究播客为例，一天的完成标准已经明确为研究报告、双人脚本、可播放音频和索引都存在。如果 Claude 只知道要写报告，而 Codex 知道还要验证音频，就会出现今天讨论的缺口。把共同完成标准放在一个 `AGENTS.md`，不同 Agent 才会对“完成”有相同理解。

第二个帮助，是 Skills 只维护一份。我们的视频生产已经涉及参考视频分析、批量生产、导出验证等多个角色。如果每个工具都有一份 Skill 副本，路径和禁用事项迟早漂移。cc-suite 的思路是保留一个真实来源，再通过明确桥接让不同工具发现它。

第三个帮助，是机器生成配置必须可识别。飞书、多维表格、MCP 和自动任务经常会修改结构化配置。若生成区块没有来源标记，下次运行就不知道能否覆盖。sentinel 区块和 provenance 文件可以直接借鉴。

[主持人]
那最小试验应该怎么做？是不是现在就把完整插件装到所有项目里？

[分析员]
不建议。最小试验应该放在一个无敏感数据的测试副本里，而且只验证两层。第一层，把共享项目规则收敛到 `AGENTS.md`，让 `CLAUDE.md` 导入它，然后分别问 Claude 和 Codex：这个项目的完成标准是什么、禁止事项是什么。看两边答案是否一致。

第二层，验证一份测试 Skill 能否同时被 `.claude/skills` 和 `.agents/skills` 正确发现。然后重复运行初始化，确认没有覆盖真实目录、没有产生重复配置、没有修改用户手写区块。

这两层通过后，才考虑 MCP 同步。双向自动实现、会话历史读取和停止前自动复核，都应该继续保持关闭，直到我们有明确用例和隐私边界。

[主持人]
为什么特别强调会话历史？

[分析员]
因为 cc-suite 注册的 Claude MCP Server 不只可以执行任务，也能让 Codex读取 Claude Code 的历史会话。默认按当前仓库限制，这还算合理；但参数 `all_projects: true` 会扩展到机器上的全部项目。读取会话意味着其中的内容会进入另一个模型服务商的上下文。

对公开代码仓库，这可能只是便利功能。对含有飞书记录、客户资料、私有路径和求职信息的工作环境，这就是明确的数据边界变化。能力存在不等于应该默认打开。我们应该把会话读取当成显式授权功能，按项目、按任务开启。

[主持人]
还有哪些容易被“多 Agent 很酷”掩盖的风险？

[分析员]
首先是循环委派。Agent A 调 Agent B，B 又通过另一条路径调回 A，如果没有深度限制和终止条件，成本、延迟和上下文会快速膨胀。其次是平台依赖。这个项目由 Shell、JavaScript 和 Python 组成，符号链接和 CLI 假设更适合 macOS 或 Linux，跨平台要单独验证。

第三是项目仍新。它没有正式 Release，维护速度很快，今天的目录和约定未来可能变化。使用时应该固定版本，不要永远追踪 main。第四是 MCP 配置里可能含本机路径和凭据，每次同步、提交前都要秘密扫描。第五是 Codex 项目需要 trust，插件 Hooks 需要功能开关；缺少这些前置条件时，文件看似生成了，运行时却可能根本没加载。

[主持人]
所以最终推荐是什么？

[分析员]
最终推荐和研究报告一致：现在就小范围使用，也就是 Use now，但只使用“共享说明和共享 Skills”这一小块，不要一开始就开启完整双向委派。

cc-suite 最值得我们的，不是让三个 Agent 互相喊话，而是建立一套不漂移的项目控制面：共享意图只有一个来源，平台差异明确保留，机器生成区块有所有权标记，重复运行不会破坏用户内容，不确定的数据宁可留空也不错误绑定。

今天最小的下一步，是选一个无敏感信息的项目副本，只验证 `AGENTS.md`、`CLAUDE.md` 和 Skills 桥接。验证通过后，再评估 MCP 同步是否真的解决了问题。会话历史读取、跨模型自动实现和停止前复核，继续要求人工显式开启。

[主持人]
好，这一期对应的仓库是 `xiaolai/cc-suite`，研究版本是 `fbcf751`。一句话收尾：多 Agent 协作的重点不是多叫几个模型，而是让它们读同一份目标、守同一条边界，并且每一次委派都能被追踪。我们下一期继续从真实项目问题出发，研究一个仓库，也交付对应的可听播客。
