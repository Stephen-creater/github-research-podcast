# 单仓库播客 02：Audicle，把你的阅读清单变成自己的播客

[主持人]
今天这一期只讲一个仓库：Audicle。它的定位很明确：一个自托管的 Podcasting 2.0 服务，把你保存的文章、PDF、DOCX、Markdown、HTML 或普通文本，变成你自己的私人播客 feed。

[分析员]
先给一句话结论：Audicle 不是“把网页念出来”的小脚本，它更像一个完整的文章转播客系统。你把 URL 或文件丢进去，后端抽取正文、清洗文本、调用 TTS、生成音频、封面和 WebVTT 字幕，最后把节目放进 RSS feed。你可以用 Pocket Casts、Overcast、Apple Podcasts 这类普通播客 App 订阅。

[主持人]
这个项目和你的需求很贴。你现在已经有 Feishu 文章树，会把 WeChat 长文、付费文章、AI 文章、投资文章剪存进去。问题是，存进去以后不代表你真的复读了。Audicle 给的思路是：把“值得保存的阅读清单”进一步变成“可以跑步听的私人播客”。

[分析员]
先讲怎么跑。README 里给的 Quickstart 很直接：克隆仓库，复制 `.env.example` 到 `.env`，然后 `docker compose up -d`。服务起来之后，Web UI 在 `http://localhost:8000/`。它启动时不强制要求你提前配好所有东西，可以在 Settings UI 里设置 LLM provider、feed 信息、管理员密码，并上传参考音色。

[主持人]
仓库结构也很清楚。`backend/` 是 FastAPI 应用和 SQLite 队列；`tts-wrapper/` 是 TTS 模型服务，使用 Chatterbox，单独容器隔离；`render/` 是页面渲染 sidecar，基于 Camoufox 和 xvfb，用来处理需要点击展开的文章；`frontend/` 是 React 和 Tailwind 的操作界面；`data/` 是运行时产物，比如数据库、MP3、封面、字幕。

[分析员]
这个架构的重点是：它把文章抽取、TTS、渲染、Feed 发布拆开了。文章抽取默认用 direct engine，加 trafilatura 抽正文。遇到复杂页面、JS 渲染、bot gate 或付费墙时，它有 fallback 机制：Googlebot、Freedium、自定义代理、Jina Reader、FlareSolverr、archive、render sidecar。它不是只假设网页都很干净。

[主持人]
这对你很关键。你经常遇到的不是普通技术博客，而是公众号、付费文章、长文、各种网页格式。Audicle 里“抽取失败不要直接生成垃圾音频”的思路值得学。它会检测短 teaser，避免把一段付费墙摘要合成 25 秒废音频。

[分析员]
再讲声音部分。Audicle 的 TTS 不是用 Mac 系统 `say`，而是 Chatterbox TTS。它支持上传参考声音，让每期用指定音色旁白。README 里说没有 GPU 也能跑 CPU，只是更慢。第一次模型下载大概 2GB，存在 `./data` volume 里的 Hugging Face cache 和 TTS home。

[主持人]
这也是它的第一个现实成本：重。它适合自托管长期服务，不适合我们现在每次都临时跑一下。你已经明确对本机重音视频任务有顾虑，所以现阶段不要贸然把 Audicle 全套跑起来。更合理的是先学习它的系统设计，把你的轻量版流水线做稳。

[分析员]
Audicle 还有几个很产品化的细节。第一，Feed 是真正的 Podcasting 2.0 feed，不是一个只能在网页播放的音频列表。第二，每期可以有 artwork，并且把封面嵌进 MP3，因为有些播放器不读 feed 图片。第三，有 pronunciation corrections，能把容易读错的词修正成你想要的读法。第四，有 webhook，生成成功或失败后可以 POST 到你的下游系统。

[主持人]
如果把它接到你的系统里，最自然的落点不是 GitHub 仓库解析，而是 Feishu 文章复读。比如你在 Feishu 的 `Information > 文章` 里挑出“AI, 未来”“认知”“投资”下面值得复读的文章，把它们导出 Markdown 或原文 URL，然后进入 Audicle 风格的队列，生成单篇文章或合集播客。

[分析员]
它和上一期讲的 gitingest 的关系也很清楚。gitingest 负责把代码仓库变成 LLM 上下文；Audicle 负责把文章或文档变成可听 feed。两个工具处理的源不同，但目标一致：把高价值信息转换成更适合你消费的形态。

[主持人]
我们再把它拆细一点。Audicle 最值得学习的不是“有一个按钮把网页变 MP3”，而是它承认内容生产是一条管线。第一步是 intake，也就是把 URL、文件、文本放进系统。第二步是 extraction，把正文、标题、作者、图片、发布时间这些元数据尽量抽出来。第三步是 normalization，把网页噪声、导航、免责声明、广告、重复段落清掉。第四步才是 TTS。第五步是 publishing，把音频、字幕、封面和 RSS 聚合在一起。

[分析员]
这对你现在很关键，因为你一开始遇到的问题其实不是“有没有声音”，而是“声音里讲的东西值不值得听”。如果上游抽取的是低质量摘要，下游声音再自然也只是把垃圾念得更好听。Audicle 的设计提醒我们：音频体验的前提是内容质量控制。比如短 teaser 检测、抽取 fallback、任务状态、失败重试，这些看起来不性感，但决定了每天自动化会不会变成每天制造废音频。

[主持人]
从文件结构看，Audicle 还把“跑模型”和“管业务”分开。主后端负责接收任务、写数据库、管理 feed；TTS wrapper 只负责把文本变成音频；render sidecar 只负责处理复杂网页。这样一个环节坏了，不会把整个系统搞成一团。你的桌面播客项目也应该学这个思想：候选仓库发现、仓库消化、写稿、TTS、验证、同步，每一步都要能单独失败、单独重跑。

[分析员]
比如今天如果 MiMo TTS 接口失败，正确行为不是重新选仓库，也不是重写稿，而是保留已经写好的 script，只把 audio 阶段标记为 failed，下一次继续合成。反过来，如果仓库 clone 失败，就不要生成一个空洞节目。Audicle 这种任务队列思路，正好可以用来约束你的“每天 5 期”：不是看起来跑过就算完成，而是 5 个 episode 都要达到 done 状态。

[主持人]
它的 RSS 思路也值得展开。为什么要做 feed，而不是只在文件夹里堆 MP3？因为 feed 代表消费端标准化。你可以用任何播客 App 订阅，手机上自动刷新、自动下载、自动记播放进度。对你这种跑步听的人来说，最终最舒服的形态不是每天去桌面找文件，而是像听普通播客一样打开播放器，里面出现“今天的 GitHub 仓库 01、02、03、04、05”。

[分析员]
不过第一版先不要急着做 feed。你现在最需要的是生产质量稳定。Feed 是分发层，质量问题在生产层。短期只要把 `outputs/` 做清楚：每期有脚本、音频、时长、仓库链接、repo commit、生成日期、是否通过校验。等这个结构连续跑几天，再把它们映射成 RSS item，就比较顺。

[主持人]
Audicle 的 iOS Shortcut 也有启发。它说明这个产品不是只给开发者用命令行，而是允许你从手机或阅读场景直接把内容送进系统。对应到你的工作流，未来可以有两个入口。一个入口是 GitHub Trending 或搜索结果自动进来；另一个入口是你在手机上看到一个仓库或文章，点分享，发到 Feishu 或快捷指令，然后进入播客候选池。

[分析员]
这就把“信息消费”变成闭环了：看到内容，丢进候选；系统判断是否值得；值得就生成音频；听完后在 daily index 里留下动作建议；如果真的能接入你的项目，再进入 Feishu 任务或项目文档。Audicle 只做了阅读清单到 feed，但它给的是一个更大的模式：输入入口标准化，产物标准化，消费入口标准化。

[主持人]
再讲一个容易被忽略的点：发音修正。技术播客里很多词很容易被 TTS 念得很难受，比如 repo 名、模型名、英文缩写、中文夹英文。如果没有 pronunciation corrections，听感会很快崩。Audicle 把这个当成产品功能，是非常对的。你的 GitHub 播客也应该维护一份词表，比如 Feishu 念“飞书”，Lark 念“拉克”或直接念英文，Codex 念“Code-ex”而不是乱读，gitingest 念成“git ingest”。

[分析员]
这一点和“有人味”直接相关。真人播客不是只靠音色自然，关键是节奏、断句、重音、术语读法和上下文解释。你之前不满意机械音，本质上是两个问题叠加：声音机械，以及文字稿像自动摘要。Audicle 给我们的提醒是，内容脚本要先像人写的，然后 TTS 才有机会像人说的。

[主持人]
如果把 Audicle 当成未来版本的蓝图，我会这样切阶段。第一阶段，桌面文件夹生成稳定的单仓库音频。第二阶段，给每期加 WebVTT 或纯文本章节，方便回看。第三阶段，生成一个本地或私有 RSS feed。第四阶段，把 Feishu 文章和 GitHub 仓库都纳入同一个队列。第五阶段，手机端一键投递内容，跑步时自动听。

[分析员]
这条路线的好处是每一步都有明确产物，不会陷入“大而全系统没跑起来”的坑。Audicle 很强，但你不需要一次复制它全部。你需要的是把它的工程原则借走：队列化、可重试、可验证、可订阅、可迁移。

[主持人]
从代码和架构层面看，Audicle 的亮点是“队列 + 后端管线 + 前端管理 + RSS 发布”。它没有把所有逻辑塞在一个脚本里。后端管理任务状态，TTS wrapper 隔离模型和 GPU/CPU 负担，render sidecar 专门解决网页复杂抽取，frontend 让操作可见。这个分层非常适合长期服务。

[分析员]
它的风险也要说清楚。第一，部署成本比普通脚本高，需要 Docker、模型下载、数据目录权限。第二，CPU 跑 TTS 会慢，GPU 才舒服。第三，处理付费文章时要注意版权和账号 cookie，cookie jar 等同账号访问权，不能随便提交或泄露。第四，如果你只是想每天 5 期 GitHub 仓库播客，Audicle 不是主引擎，它更适合作为“文章转播客”支线。

[主持人]
那你现在应该怎么用这期信息？我建议把 Audicle 放进路线图，不立刻重部署。短期我们继续用桌面 `GitHub播客` 文件夹和词元跳动 MiMo TTS 生成音频。中期可以借鉴 Audicle 的目录结构，把每期产物做成：原始源、清洗稿、音频、字幕、manifest。长期再考虑真正做 RSS feed，让你在播客 App 里订阅自己的节目。

[分析员]
如果要给这个仓库一个 fit 评分，我会给高分，但不是因为它能马上替代当前脚本，而是因为它代表了最终形态：你的阅读清单不是收藏夹，而是一个你拥有的播客系统。你不再被 B 站、YouTube、公众号的播放形态绑架，而是把内容拿回自己的 feed。

[主持人]
这一期总结：Audicle 解决的问题是“我读太多，想把有价值的阅读清单变成自己拥有的音频 feed”。它的核心架构是 FastAPI 后端、SQLite 任务队列、Chatterbox TTS wrapper、网页渲染 sidecar、React 前端和 Podcast RSS。对你来说，它最值得借鉴的是：feed 所有权、失败可重试、抽取质量控制、音频产物管理。

[分析员]
下一步动作很简单：不要现在重部署 Audicle。先把你的桌面播客项目按 Audicle 的产物思想改进，每期生成 script、audio、manifest、repo notes。等单仓库播客稳定跑满几天，再考虑把它升级成真正的私有 RSS feed。
