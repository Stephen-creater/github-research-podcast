# 单仓库播客 05：feishu-pm-kit，把飞书 AI 项目经理产品化

[主持人]
今天这一期只讲一个仓库：feishu-pm-kit。它不是播客工具，而是一个飞书 AI 项目经理 Agent 加项目管理系统的部署套件。它的价值在于，把 Claude Code、CLAUDE.md 操作手册、lark-cli 飞书通道和项目档案治理组合成一个可复用模板。

[分析员]
一句话结论：feishu-pm-kit 适合你理解“怎么把一次性 AI 工作流产品化”。它不是只做一个 bot，而是把团队实例、配置、项目档案、索引、健康检查、多维表格同步、日报和风险治理都放进同一套结构里。

[主持人]
先讲它解决什么问题。很多人做 AI agent，是在一个群里加一个机器人，然后靠 prompt 维持行为。问题是换一个团队就要重写一套，状态和项目资料也散落在聊天里。feishu-pm-kit 的做法是：共享引擎一份，团队实例多份。新团队只加 config，再跑 bootstrap，就能上线。

[分析员]
仓库结构非常清楚。`engine/` 是共享引擎，里面有 `gen_index.py`、`feishu.py`、`bitable_sync.py`、`bootstrap.py`、`health.py`。`templates/` 是实例脚手架，包括 config 示例、CLAUDE.md 模板、项目档案模板。`instances/` 放各团队实例。`tests/` 是 pytest 测试。`docs/` 是设计文档和部署清单。

[主持人]
这个结构对你很有启发。你的 GitHub 播客项目现在也要避免变成散乱脚本。它也应该分成共享引擎和 episode 实例。共享引擎负责搜索、读仓库、写稿、TTS、验证、commit push；episode 实例保存每期的 repo、commit、script、audio、duration、status。

[分析员]
feishu-pm-kit 的核心理念是：项目档案用 Markdown frontmatter 作为唯一事实源。脚本根据这些 Markdown 生成索引和健康看板，再做四项治理校验，还可以单向镜像到飞书多维表格。这个设计比直接在多维表格里改来改去更可控，因为源文件可版本管理。

[主持人]
四项治理校验很值得讲。第一，必填和占位，检查必填字段是否齐全，是否还写着待补充。第二，公式和一致性，比如毛利等于收入减成本，风险计数和表格行数一致。第三，新鲜度，进行中项目超过阈值没更新就报警。第四，命名和编号，检查编号格式、文件名匹配、重复和旧版本残留。

[分析员]
这和你的每天 5 期播客很像。你也需要治理校验：当天是否满 5 期；每期是否只有一个仓库；是否有脚本和音频；音频是否 10 到 40 分钟；是否通过解码；是否 commit push；是否避免重复仓库；是否有 repo commit。没有这些校验，自动化很快会劣化。

[主持人]
快速开始也说明它偏工程化。先装 Python 环境，安装 requirements，跑 pytest。部署新团队实例时，阶段 0 是手动飞书后台建应用和授权；阶段 1 跑 `python3 -m engine.bootstrap --name team-alpha`；阶段 2 进入实例目录启动 Claude Code。

[分析员]
它的常用命令也很像一个操作系统：`engine.gen_index` 生成索引和校验；`engine.feishu` 做 poll、send、ws-start；`engine.bitable_sync` 推飞书多维表格；`engine.health` 做健康检查和自动重启。每个命令都有明确边界。

[主持人]
对你来说，feishu-pm-kit 最值得吸收的是“实例化”和“治理”。你不是只要今天手工生成 5 个音频，而是要一个每天稳定生产 5 期的系统。那就必须有 daily instance。比如 `outputs/daily/2026-07-05.md` 是当天实例，里面列出 5 个 episode，每个 episode 有状态。

[分析员]
这个仓库还体现了一个更大的能力：AI FDE 或 AI 产品化不是会用工具，而是能把工作流变成可部署、可测试、可迁移、可治理的系统。你未来讲自己的案例时，不能只说“我让 Codex 生成了播客”，要说“我建立了一个单仓库播客生产系统，有源筛选、每日配额、音频合成、质量校验、远程同步和后续 Feishu 控制面”。

[主持人]
它也有局限。第一，它依赖飞书应用权限和 lark-cli，部署门槛不低。第二，当前 README 里真实实例迁移和多维表格联调还是 Phase 5，依赖飞书侧权限。第三，它是项目管理系统，不是内容生产系统，不能直接拿来生成播客。第四，如果多个 Claude Code 实例常驻，要注意 CPU、内存和 WebSocket 资源。

[分析员]
但作为架构参考，它很强。它告诉你：不要把一切逻辑藏在聊天里。要有模板、有实例、有索引、有校验、有健康检查、有同步。你的桌面播客项目也应该这样成长。

[主持人]
短期怎么落地？第一，把每天 5 期写进自动化。第二，创建 daily index。第三，创建 episode manifest。第四，给脚本增加 keychain 读取，避免 key 落盘。第五，每天结束时 commit push，远程仓库就是事实记录。

[分析员]
中期可以进一步借鉴它的 Bitable 同步。比如每天 5 期生成后，把 episode metadata 推到飞书多维表格：仓库名、URL、时长、状态、fit 方向、是否已听、下一步动作。这样播客不只是文件，而是进入你的管理系统。

[主持人]
我们把 feishu-pm-kit 的治理思想再落到今天这个播客项目。你刚刚说“每天至少更新 5 个播客”，这句话如果只写在聊天里，很快就会失效。系统必须能回答几个问题：今天是哪一天；今天目标是几期；已经完成几期；哪几期失败；失败卡在哪个阶段；远程仓库是否已经同步；音频是否真的能播放；是否每期只对应一个仓库。

[分析员]
feishu-pm-kit 给的答案是：不要靠记忆，靠事实源。它用 Markdown frontmatter 和生成索引来管理项目档案。你的播客系统也可以用 daily index 做事实源。比如 `outputs/2026-07-05_daily_index.md` 里列出 5 条，每条包含 repo、remote、repo commit、script path、audio path、duration、status、fit reason。自动化每天先读这个文件，发现缺口再补。

[主持人]
这比“每天跑一次脚本”强得多。因为跑脚本只说明动作发生了，不说明结果合格。治理文件说明结果是否达标。比如今天有 5 个 m4a 文件，但其中一个只有 6 分钟，那不算完成；一个音频通过生成但 ffmpeg 解码失败，也不算完成；一个文字稿讲了三个仓库，也不算完成。

[分析员]
feishu-pm-kit 的四类校验可以直接迁移。必填校验变成：每期必须有仓库名、URL、脚本、音频、时长、状态。公式校验变成：完成数等于 done 状态条目数，不能手写虚假数字。新鲜度校验变成：每天都要有当天 index，不允许拿昨天的 5 期冒充今天。命名校验变成：文件名必须包含日期和仓库名，避免堆成一堆乱文件。

[主持人]
这也能帮你之后讲成一个作品。很多人说自己做了 AI 自动化，只能展示一段聊天记录。你的版本如果做成 daily index、远程 commit、音频产物、自动化记录、质量校验，就能说明你不是在玩 prompt，而是在搭一个可运行的信息产品化系统。

[分析员]
再看 feishu-pm-kit 的 engine/instances 分层。它把共享能力放在 engine，把具体团队放在 instances。你的播客系统也可以类似：engine 里是抓仓库、读 README、生成脚本、合成音频、验证和同步；outputs 里是每天的具体产物。以后如果你要做“GitHub 仓库播客”“Feishu 文章播客”“训练营课程复读播客”，它们就可以共用 engine，但有不同 instance 配置。

[主持人]
这就是可迁移性。今天你先做 GitHub 仓库，明天可以换成公众号文章，后天可以换成课程录播笔记。如果没有这种分层，每换一个内容源就会重写一套脚本。feishu-pm-kit 的模板化思想提醒我们：不要把流程绑死在一个场景里。

[分析员]
它的测试目录也值得注意。很多个人自动化失败，是因为没有测试意识。你不需要给播客文字内容写单元测试，但关键规则要能检查：解析脚本是否能得到主持人和分析员段落；输出音频是否在 10 到 40 分钟；daily index 是否满 5；密钥扫描是否没有泄露；git 是否成功 push。这些都可以变成小的验证步骤。

[主持人]
这里还有一个现实问题：每天 5 期会产生大量文件。按每期 10 到 20 分钟，一个音频十几 MB 到几十 MB，一周就是几百 MB。远程仓库长期存 m4a 会膨胀。第一阶段可以接受，因为你需要看到产物；中期要考虑 Git LFS、对象存储、或只把脚本和 manifest 入 git，音频放专门存储。

[分析员]
但你刚才明确要求每次本地和远程 commit push 同步，所以现在我会先按这个执行。只是 daily index 里要把产物列清楚，后面如果仓库变大，再迁移存储策略。这个边界要早知道，不要等仓库几 GB 了才处理。

[主持人]
feishu-pm-kit 还有一个健康检查思路。它不仅生成项目资料，还能检查服务状态和自动重启。你的播客自动化也需要健康视角：今天是否运行；是否有 5 期；失败原因是什么；最近一次 push 是哪个 commit；TTS 是否可用；是否找不到 key；是否因为网络失败中断。

[分析员]
以后可以把这些健康信息回传到 Feishu。比如每天早上发一条：今日目标 5 期，已完成 5 期，总时长 72 分钟，远程同步 commit 某某，失败 0。这样你不用进文件夹检查，也能知道系统还活着。

[主持人]
如果和前面几个仓库连起来看，路径就清楚了。gitingest 提供仓库到上下文的压缩思路；Audicle 提供内容到私有 feed 的产物思路；ai-podcast-studio 提供每日 pipeline 和 quota 思路；feishu-claude-code 提供 Feishu 遥控思路；feishu-pm-kit 提供治理和实例化思路。这 5 期不是随机选的，它们合起来就是你的 GitHub 播客系统路线图。

[分析员]
这也是为什么每个仓库要单独讲。单独讲完之后，我们不只是知道“有这些工具”，还知道每个工具在你的系统里扮演什么角色。gitingest 是消化器，Audicle 是长期 feed 蓝图，ai-podcast-studio 是生产线，feishu-claude-code 是控制面，feishu-pm-kit 是治理框架。

[主持人]
短期执行上，feishu-pm-kit 对今天的要求很简单：别只生成文件，要生成可检查的 daily index；别只本地保存，要 commit push；别只写“完成”，要验证音频时长和解码；别把 key 写进仓库；别把第三方 clone 目录提交。这样每天 5 期才有工程上的可信度。

[分析员]
这期也给你未来面试或 AI FDE 案例一个表达方式。你可以说：我把 GitHub Trending 和个人项目上下文结合，建立了一个每日单仓库播客系统。它每天生成至少 5 期，每期 10 到 40 分钟，包含仓库架构、适配分析和落地建议；系统有 TTS、质量校验、版本同步、daily manifest，后续可接 Feishu 控制面和 RSS feed。这个表达比“我让 AI 读 GitHub 仓库”强很多。

[主持人]
这期总结：feishu-pm-kit 解决的是“把飞书 AI 项目经理模式产品化”的问题。它的核心设计是共享 engine、多实例 config、Markdown frontmatter 事实源、索引和四项治理、飞书通道、多维表格镜像、健康检查。对你的价值是，把播客自动化升级成可治理系统。

[分析员]
下一步动作：用它的治理思想改造 `GitHub播客`。每天 5 期不是口号，而是一个可检查的 daily target。今天先补齐 5 期，后面自动化每天继续跑。
