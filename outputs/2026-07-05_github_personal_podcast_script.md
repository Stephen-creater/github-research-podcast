# GitHub 到跑步播客第一期：把低信息量内容换成你的私人项目雷达

[主持人]
今天这期，是一个很具体的实验。起点不是“再做一个 AI 新闻摘要”，也不是把 GitHub Trending 机械念一遍。真正的问题是：你已经把每天锻炼这件事养起来了，但锻炼时听什么，决定了这半小时到底是在消遣，还是在给未来的项目和判断力补燃料。

[分析员]
对。你前面提到的那个判断很准：很多二三十分钟的视频讲解，偶尔当乐子没问题，但如果它的信息密度低、结构浅、和你正在做的事情没有连接，那它就只适合当背景噪音。它不配占用你每天最稳定的一段注意力。

[主持人]
所以我们今天不做泛泛的“开源项目推荐”。我会把这一期当成你的第一条私人 GitHub 播客样品：我已经从 GitHub 上抓了几个和你最贴合的项目，读了 README 和目录结构，然后围绕你的上下文来讲：它们是什么，解决什么问题，和你的 Feishu、Codex、知识库、AI FDE、AI HOT 这些线怎么接上，以及下一步该怎么自动化。

[分析员]
还有一个现实边界先说清楚：Codex 这个环境本身没有直接给我一个高质量音频生成按钮。第一版音频用的是你这台 Mac 自带的中文语音合成，再用 ffmpeg 封装成 m4a。它不需要 ElevenLabs，不需要额外买 TTS，也不需要你再提供国内模型接口。缺点也明确：声音自然度不如商业模型，但足够验证链路。

[主持人]
这一期选了六个仓库。第一，gitingest。第二，ai-podcast-studio。第三，Audicle。第四，feishu-claude-code。第五，feishu-pm-kit。第六，Morning-news-podcast。它们不是同一类项目，但刚好覆盖了你要的全链路：抓仓库、压上下文、选内容、写播客、合成音频、推送到你真正工作的地方。

[分析员]
如果只选一个最底层的积木，那一定是 gitingest。它做的事情很简单：把一个 Git 仓库，变成适合丢给大模型读的文本上下文。它支持命令行，也支持 Python 包；可以读本地目录，也可以读 GitHub URL；还会输出目录结构、文件内容、大小和 token 级别的统计。

[主持人]
它的重要性在于，仓库不是文章。一个仓库有 README，有源代码，有配置，有测试，有历史意图。你如果只让 AI 看 README，它最多能讲产品介绍；但你要做的是“仓库转播客”，就必须先把仓库压成一个可推理的上下文。gitingest 正好是这个入口。

[分析员]
放进你的流程里，它的位置大概是这样：夜间脚本先抓 GitHub Trending 或者你订阅的源，比如 agent、Feishu、TTS、RSS、LLM tooling、automation。然后对每个候选仓库跑 gitingest，把结果喂给 Codex，让 Codex做三件事：第一，判断这个仓库和你是否真的相关；第二，读出它的架构和关键设计；第三，把它翻译成你听得懂、且能落到项目的播客脚本。

[主持人]
这里有个关键差异：你不是一个从零学编程的人，也不是一个单纯追热点的人。你现在的主线更像是：个人知识库、飞书工作流、AI 信息雷达、AI FDE 求职和案例包装、以及把 Codex 当作执行合作者。所以仓库筛选不能只看 star 数，而要看它能不能嵌进这些现有系统。

[分析员]
这就是为什么第二个仓库 ai-podcast-studio 很值得讲。它几乎就是你描述的“夜间自动生成播客”的完整产品形态。它可以监控网站、RSS 和 X 账号，把新内容收集起来，按相关性打分，筛掉不值得讲的东西，再写成脚本，生成音频，最后发布成 podcast RSS。你可以在普通播客 App 里订阅。

[主持人]
它的架构不是玩具。它有源监控，有去重，有相关性评分，有脚本生成，有历史节目检索，避免重复讲旧话；它还支持 review，也就是先让你审稿，再花钱生成音频；并且它会记录失败阶段，某一步挂了可以从失败处继续，不用把前面的 LLM 调用重新付一遍。

[分析员]
这点对你很关键。你之前已经有 AI HOT 的需求，但你其实不满足于“别人整理好的 AI 热点”。你想要的是低噪声、自己拥有源管线、能和自己项目发生关系的版本。ai-podcast-studio 给的启发是：不要把这个东西做成“新闻 App”，而要做成“源到判断的流水线”。

[主持人]
如果按你的版本来配，它的主题不应该叫 AI News，而应该叫“叶耀楠的项目雷达”。子主题可以分成五类：第一，Coding Agent 和 Codex 生态；第二，Feishu 和知识库自动化；第三，TTS、播客、阅读转音频；第四，AI 产品化和 FDE 案例；第五，赚钱、投资、信息效率相关但必须降噪。

[分析员]
它的缺点也要说清楚。这个项目的完整音频层主要面向 OpenAI、ElevenLabs、Inworld 这类 TTS provider。它可以用 Ollama 覆盖一部分 LLM，但声音这一层如果追求质量，还是需要外部 TTS。也就是说，它适合作为“产品架构参考”，但不是今天零成本第一版的直接运行方案。

[主持人]
第三个仓库 Audicle，和 ai-podcast-studio 有点像，但它的出发点更贴近“阅读清单变成播客”。它支持粘贴 URL，也支持上传 PDF、DOCX、Markdown、文本、HTML，然后把内容变成私有播客 feed。它强调你拥有自己的 feed，不把音频锁在平台里。

[分析员]
这和你的 Feishu 文章剪存流程很贴。你现在会从 WeChat、长文、付费文章、AI 文章、投资文章里筛材料，剪存进 Feishu 的 Information 文章树，再做分类、复读、提炼。Audicle 给了一个下一步想象：那些已经进入“值得保存”的文章，不一定都要坐下来读第二遍。有些可以生成音频，变成跑步时的复读材料。

[主持人]
Audicle 比较强的一点是它对文章抽取、付费墙、网页渲染、音频质量检查、RSS、封面、WebVTT transcript 都想得比较完整。它不是简单读网页，而是像一个私有 Podcasting 2.0 服务。它甚至有 iOS Shortcut 的思路：看到一篇文章，丢进去，处理完以后自然出现在你的播客 feed 里。

[分析员]
但它也有成本。它的本地 TTS 依赖 Chatterbox，第一次模型下载大概是 GB 级，CPU 也能跑但会慢。你之前对本地音视频处理造成机器负载很敏感，所以不能盲目把这个跑成常驻重任务。更合理的方式是：先把它当成“长期形态参考”，现在用轻量脚本和 Mac 自带 TTS 验证需求。

[主持人]
接下来两个仓库，是你和别人最大的差异点：Feishu。第四个是 feishu-claude-code。它的描述非常直接：飞书是遥控器，Claude Code 是在你电脑上真实干活的那只手。出门在外发一条飞书消息，电脑那边就开始跑代码、改文件、读文档。

[分析员]
这个项目支持文本、富文本、图片、文件、语音消息、转发消息；还能自动解析飞书文档链接，长回复自动写入飞书文档，再把链接发回聊天。它还支持 session 管理、工作目录绑定、macOS launchd 保活。对你来说，这不是“又一个聊天机器人”，而是把你现在的 Feishu 中枢和本机 Codex 执行能力接起来。

[主持人]
想象一下这个播客系统成熟后的操作方式。你在 Feishu 的 Daily 里写一句：“今晚跑步前生成一期 GitHub 项目雷达，重点看 Feishu 自动化、Agent 工具和 TTS。”桥接器在本机收到消息，触发 Codex 或本地脚本，抓仓库、写脚本、合成音频、把 m4a 回传飞书，再把摘要写到知识库。

[分析员]
这条路很有吸引力，但权限边界必须很硬。飞书 Bot 会读你发给它的文档链接，长回复还可能生成飞书文档。它要求 App ID、App Secret、lark-cli 登录、docx 和 drive 权限。对你这种把知识库当长期资产的人来说，不能一上来就全量接入。应该从一个单独的测试群、单独的工作目录、单独的私有仓库开始。

[主持人]
第五个仓库 feishu-pm-kit，则更像“把项目管理本身产品化”。它把 Claude Code、CLAUDE.md 操作手册、lark-cli 飞书通道组合起来，做成 AI 项目经理 Agent。项目档案用 Markdown frontmatter 当事实源，再生成索引、健康看板和治理校验，还可以单向镜像到飞书多维表格。

[分析员]
这对你的 AI FDE 和求职案例很重要。你现在不是只想“会用 AI 工具”，而是要能证明自己可以把场景产品化。feishu-pm-kit 给你的启发是：项目不是靠聊天记录管理的，项目要有事实源、索引、状态、风险、日报、验收标准。你之前做 UML 刷题站、AI HOT 监控、Feishu 剪存、简历案例，都可以被整理成这种可展示的项目档案。

[主持人]
这和播客有什么关系？关系在于播客不应该只是“听完就算”。每期播客都应该落一个小状态：这个仓库是什么，和我哪个项目有关，下一步动作是什么，是否进入关注列表，是否创建任务，是否写入 Feishu。feishu-pm-kit 提醒我们：音频只是入口，后面要接项目治理。

[分析员]
第六个仓库 Morning-news-podcast 是 Android 原生应用，做每日新闻音频简报。它支持多源新闻聚合，LLM 生成播报文稿，TTS 合成，定时任务，播放器，倍速，日志。它不是最适合你当前 Mac + Codex + Feishu 的主线，但它有两个参考价值。

[主持人]
第一个参考价值，是它更贴近国内模型和手机端。它提到 DeepSeek、阶跃星辰、通义千问、小米 MiMo TTS，以及 OpenAI 兼容协议。这说明如果未来你真的想接国内模型，技术上不是问题。第二个参考价值，是它把“生成”和“收听”放在同一个移动端体验里。你跑步时真正接触到的不是脚本，而是播放器和倍速体验。

[分析员]
现在把六个项目合起来，就能得到一个最小可行架构。第一层，源发现。GitHub Trending、GitHub search、AI HOT selected、你关注的开发者、Feishu 文章收件箱，都可以进入候选池。第二层，内容压缩。仓库用 gitingest，文章用 trafilatura 或 Audicle 那类抽取器，Feishu 文档用 lark-cli。第三层，个性化筛选。不是按热度，而是按“和你当前项目的连接强度”打分。

[主持人]
第四层，脚本生成。这里 Codex 很适合做第一版，因为它已经有你的上下文，也能读本地仓库。脚本不是总结 README，而是回答四个问题：这个项目解决了什么真实问题；架构上有什么值得偷师；它和你正在做的事情如何结合；下一步你应该做什么。第五层，音频。短期用 Mac TTS 零成本验证，长期可以换成 Audicle 的本地 TTS，或者国内 TTS，或者 ElevenLabs。

[分析员]
第六层，存储和同步。这一点你刚刚补充得很对。这个系统不能只散落在本地临时目录。每一期都要有本地文件和远程仓库同步：选题清单、仓库 manifest、脚本、音频、生成时间、用到的 commit，都要 commit 再 push。这样它不是一次性输出，而是你的个人知识生产流水账。

[主持人]
这一点我已经按私有仓库的方式处理：本地会保留生成脚本、项目清单、播客稿、m4a 音频；第三方仓库源码只放在 work 区，不提交；临时合成片段也不提交。因为你要的是自己的 pipeline，不是把别人仓库复制一份进你的仓库。

[分析员]
如果我们给这个系统取一个名字，它不应该叫“GitHub Trending 播客”。更准确的名字是“个人项目雷达”。它每天不是问“今天什么火”，而是问“今天有什么东西，能让我的知识库更干净、让我的项目更自动、让我的求职案例更强、让我的运动时间更有复利”。

[主持人]
按这个标准，今天的优先级我会这样排。第一优先级，保留 gitingest 思路，把仓库转上下文做成固定步骤。第二优先级，用 ai-podcast-studio 的架构设计作为长期蓝图，但不要现在就重部署一套重系统。第三优先级，用 Audicle 的“阅读清单到私有播客”思路，未来对接你的 Feishu 文章树。第四优先级，小范围测试 feishu-claude-code，让 Feishu 能触发本机任务。第五优先级，把关键项目按 feishu-pm-kit 的项目档案方式沉淀。

[分析员]
落到接下来七十二小时，建议只有三件事。第一，先听完今天这条样品音频，判断你能不能接受运动时听这种密度。如果声音太机械，但内容有用，那下一步才值得换 TTS。第二，把每天或每六小时跑一次的自动化先设置为“生成草稿和清单”，不要一上来全自动发音频，避免低质量内容污染你的 feed。第三，选十到二十个源，而不是只依赖 Trending。

[主持人]
这些源可以包括：GitHub 上 agent、codex、lark、feishu、tts、podcast、rss、knowledge management、browser automation 相关关键词；AI HOT selected；你 Feishu 里正在完善的 AI 未来、赚钱、投资、认知文章目录；以及你正在准备的 AI FDE、Superun、杭州有赞这一类机会线。

[分析员]
筛选规则要非常苛刻。一个项目如果只是“看起来很 AI”，但不能进入你的系统，就降权。一个项目如果 star 少，但刚好能解决你 Feishu、Codex、知识库、跑步听播客之间的摩擦，就升权。你要训练的不是信息获取，而是信息进入项目的能力。

[主持人]
最后回到你开头说的那个时代里程碑。真正重要的不是“AI 能生成播客”这件事。生成音频本身会越来越便宜。真正重要的是，AI 能把每天涌现出来的开源项目、文章、工具、团队历史，和你的个人处境连接起来，讲成你能在运动中吸收的判断。

[分析员]
也就是说，未来的学习不是你去找内容，而是内容经过你的历史、项目、认知水平和下一步目标过滤后，主动变成适合你的形态。视频、文章、仓库、文档、聊天记录，都只是原材料。真正的产品是“个人上下文参与过的解释”。

[主持人]
这就是今天第一版要验证的东西：不是让你听更多内容，而是让你每天那段本来可能被低信息量视频占掉的运动时间，变成一个私人项目雷达。听完以后，如果你只记住一句话，就记住这句：GitHub Trending 不重要，和你项目发生关系的 GitHub Trending 才重要。

[分析员]
下一次迭代，可以做三种升级。第一，接入更自然的 TTS，让它真的像播客。第二，把候选源扩展到 Feishu 文章和 AI HOT selected。第三，让每期结尾自动生成行动项，写进远程仓库，也可以写回 Feishu。这样音频就不只是被听过，而是进入你的执行系统。

[主持人]
第一期到这里。今天的结论很明确：先别追求完美声音，也别急着上重系统。先把“选仓库、读仓库、按你的上下文写脚本、生成可听音频、commit、push”这条闭环跑通。闭环跑通之后，声音、来源、调度、Feishu 回写，都可以逐层升级。
