# 单仓库播客 04：feishu-claude-code，把飞书变成 Claude Code 遥控器

[主持人]
今天这一期只讲一个仓库：feishu-claude-code。它的定位非常直白：飞书是遥控器，Claude Code 是在你电脑上真正干活的手。你在飞书里发消息，家里或桌面的电脑就能跑代码、读文件、改项目、回传结果。

[分析员]
这和你自己的工作方式高度相关。你的真实操作面不是单一 IDE，而是 Feishu、Codex、知识库、微信文章、简历和项目文档混在一起。feishu-claude-code 的价值在于，它把 Feishu 从“资料存放处”升级成“任务触发入口”。

[主持人]
先讲它解决什么问题。普通 Claude Code 需要你坐在电脑前输入任务。普通飞书机器人只能聊天，很难真正操作本地文件。这个项目把两者接起来：飞书 App 通过 WebSocket 收消息，本地 Python bridge 调用 Claude CLI，Claude 真的在本机工作，再把结果流式返回飞书。

[分析员]
README 里列的能力挺具体。文本和图片消息支持，富文本和转发消息支持，PDF、Word、Excel、压缩包等文件消息可以自动下载后分析，语音消息可以下载到本地，飞书文档链接可以自动解析并嵌入上下文，长回复可以自动生成飞书文档，聊天里只发链接。

[主持人]
这对你的播客系统很有想象空间。以后你可以在飞书里说：“今天给我生成 5 期 GitHub 单仓库播客，优先 Feishu 自动化和 TTS 项目。”桌面机器收到后，进入仓库根目录，跑候选搜索、写脚本、合成音频、commit push，然后把音频链接或文件回传飞书。

[分析员]
架构上它分三层。第一层是飞书 App，用长连接收发消息。第二层是 `feishu-bridge`，也就是本地 Python 进程，核心入口是 `main.py`。第三层是 Claude CLI，通过 subprocess 方式启动。旁边还有 `lark-cli`，负责文档、文件、IM 操作，尤其是以用户身份读写飞书文档。

[主持人]
它的关键设计决策也写得很清楚。收消息走 lark-cli WebSocket，不暴露公网端口；发消息走 Python SDK，用租户 token 做流式卡片 patch；文档操作走 lark-cli，用用户 token 直接读写文档。这个设计把消息通道和文档通道分开了。

[分析员]
安装并不轻。你需要 Python 3.11 以上、Claude Code CLI、Claude Pro 或 Max 订阅、lark-cli 1.0.20 以上。飞书开放平台要创建企业自建应用，添加机器人能力，开启消息、资源、文档、Drive 上传等权限，事件回调用长连接接收 `im.message.receive_v1`。

[主持人]
这也说明它不是玩具脚本。它触碰你的本机执行权限、飞书文档权限、文件上传权限。对于你来说，最重要的不是“能不能跑”，而是安全边界怎么设。不能一开始就给它整个 home 目录和所有知识库权限。

[分析员]
它的命令系统很适合你。`/new` 开新 session，`/resume` 恢复历史 session，`/stop` 停止当前任务，`/status` 看状态。`/cd` 切换工作目录，`/ws save` 保存命名工作空间，`/ws use` 把当前群或私聊绑定到一个工作目录。这对桌面 `GitHub播客` 文件夹非常合适。

[主持人]
如果你要真接入，我建议只建一个单独的飞书测试群，名字就叫 GitHub 播客雷达。只把这个群绑定到仓库根目录。机器人只在这个目录里工作，不许默认进入 home，不许读其他文档，除非你显式发链接。

[分析员]
这个项目还有文件回传能力。Claude 生成的文件可以通过 `[[SEND_FILE:./report.md]]` 上传到飞书 Drive，再返回下载链接。对于你的播客系统，可以把当天的 m4a、script、daily index 作为文件回传，或者只回传远程仓库链接。

[主持人]
但也有几个坑。第一，飞书 Bot 自动读取你发的文档链接，群聊中不能随便丢敏感文档。第二，App ID 和 Secret、lark-cli 用户 token、Claude CLI 权限都要分开存储。第三，长连接和本机休眠会引入稳定性问题，虽然 README 说它支持休眠唤醒自动重连。第四，后台保活在 macOS 要用 launchd，配置错路径会启动失败。

[分析员]
和你的“每天 5 期”需求结合起来，这个项目不是生成器，而是控制面。真正生成还是桌面 `GitHub播客` 项目做；feishu-claude-code 负责让你在飞书里触发、暂停、查看状态、拿文件。它是遥控器，不是播客引擎。

[主持人]
它对你个人系统的价值非常大，因为你的 Feishu 不是普通聊天工具，而是知识库、Daily、项目、文章、习惯账户、简历材料的集合。把 Codex 执行能力接进 Feishu，等于让你的知识库从“被动保存”变成“主动执行”。

[分析员]
短期不建议马上部署全量。更稳的路径是：第一，先把桌面播客项目独立跑稳，每天 5 期能正常生成。第二，建测试飞书应用和测试群。第三，只绑定 `GitHub播客` 目录。第四，给机器人固定命令，比如“今日状态”“补齐今日 5 期”“重新生成某一期音频”“列出候选仓库”。

[主持人]
我们再更具体一点，把它当成你未来的播客控制台来看。现在你是在 Codex 桌面里直接说“继续”，系统在桌面文件夹里工作。这个模式能跑第一版，但有一个限制：你必须打开这个上下文，并且自己追问状态。feishu-claude-code 的意义，是让触发入口回到你最常用的 Feishu 里。

[分析员]
它可以形成四类命令。第一类是状态命令，比如“今天完成几期了”“哪一期失败了”“音频时长是多少”。第二类是生产命令，比如“补齐今天 5 期”“重新合成 Audicle 的音频”“把某个仓库加入候选”。第三类是治理命令，比如“列出本周重复讲过的仓库”“显示低于 10 分钟的音频”。第四类是分发命令，比如“把今天 5 个 m4a 发到群里”或“生成今日索引文档”。

[主持人]
这些命令并不需要一开始都实现，但它们说明了这个仓库的价值：把本地执行能力包装成聊天入口。你的 Feishu 本来就存着 Daily、文章、项目、简历、训练营材料。未来如果播客系统也能被 Feishu 控制，你就不用在不同工具之间来回切了。

[分析员]
不过这里一定要讲安全，因为 feishu-claude-code 的能力很强。它可以让 Claude Code 在本地运行，意味着错误配置会放大风险。对你的版本，最重要的边界是工作目录白名单。机器人只能进入仓库根目录，不默认读桌面其他文件，不默认读 Downloads，不默认读整个 Feishu 知识库。你需要它处理哪个文档，就显式发链接。

[主持人]
第二个边界是命令白名单。不是任何飞书消息都直接交给 Claude Code 执行。至少在第一版里，应该只接受几种固定意图：查看状态、补齐今日、重跑某期、列候选、发文件。真正开放式的“帮我做任何事”，要等安全策略、日志和权限都成熟之后再开。

[分析员]
第三个边界是密钥管理。这个项目会涉及飞书 App Secret、lark-cli 用户授权、Claude CLI 登录状态。你的播客系统还涉及 TokenDance TTS key。所有这些都不能出现在仓库里。正确做法是 Keychain、环境变量或本机私有配置，远程仓库只保存代码、脚本、产物和不敏感 manifest。

[主持人]
从代码层面看，feishu-claude-code 的 `main.py`、`commands.py`、`claude_runner.py`、`session_store.py` 这些文件名已经说明了它的核心机制。main 负责启动桥；commands 负责解析聊天命令；claude_runner 负责实际启动 Claude；session_store 负责会话状态。它不是一个单纯 webhook，而是一个有会话、有工作区、有并发控制的本地服务。

[分析员]
这个会话设计对你也有用。比如“GitHub 播客雷达”这个飞书群可以绑定一个固定 workspace，也就是桌面 `GitHub播客`。群里每次发命令，不需要重复说明路径。更进一步，另一个群可以绑定“简历案例包装”，另一个群绑定“Feishu 知识库整理”。这就从一个机器人变成多个工作空间的入口。

[主持人]
但今天这一期只讲它和播客的关系。最直接的集成路径是：每天早上自动化跑完 5 期后，把 daily index 推到 Feishu 群；如果某期失败，群里发失败原因；你在群里回复“重跑 03”，机器人就只重跑第三期。这样你不需要打开终端，也不需要进 GitHub 仓库看文件。

[分析员]
还有一个非常适合你的点：长回复转飞书文档。播客稿件通常很长，直接发群会刷屏。feishu-claude-code 可以把长回复生成文档，再发链接。这意味着每日 5 期的文字稿、仓库分析、候选列表，都可以进入你的 Feishu 知识库，而不是散在聊天里。

[主持人]
如果要做成真正可用的系统，我会分三步。第一步，不接 Claude Code，只用普通 Codex 自动化每天在桌面生成文件并 push。第二步，让 Feishu 只接收状态和文件链接，不允许它触发本地执行。第三步，再启用受限命令，比如补齐、重跑、列候选。这样权限是逐步打开的。

[分析员]
这也符合工程常识：控制面要比生产面晚一点接入。生产面是生成播客的能力，今天必须先跑稳。控制面是远程触发和查看，跑不稳时接入只会让问题更难定位。feishu-claude-code 很有价值，但顺序不能反。

[主持人]
从产品角度看，这个仓库其实启发了一个很强的个人工作台形态。你不再打开十个工具找文件，而是在 Feishu 里对自己的本地 AI 工厂下指令。GitHub 仓库播客只是第一个应用，后面还可以是文章复读、简历案例压缩、项目日报、训练营资料整理。

[分析员]
所以我会给它的定位写得很清楚：它不是今天的 TTS 引擎，不是仓库筛选器，也不是 RSS 生成器。它是未来把你的自动化暴露给 Feishu 的遥控层。现在先收藏架构，等播客日更系统稳定后再接。

[主持人]
这期总结：feishu-claude-code 解决的是“如何用飞书远程控制本机 Claude Code 工作”的问题。它的核心架构是飞书 WebSocket、本地 Python bridge、Claude CLI、lark-cli 文档通道。它适合做你的播客系统控制面，但必须严格限制目录和权限。

[分析员]
下一步动作：现在不急着部署，但把它列为后续控制面候选。等每天 5 期的桌面自动化稳定后，再用这个仓库把 Feishu 接进来，让你不用打开 Codex 也能触发和查看播客生产。
